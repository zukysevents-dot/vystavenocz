## Kontext pro AI agenty

`AGENTS.md` je hlavní společný kontext pro Codex a ostatní AI coding agenty. `CLAUDE.md` je Claude Code vstup se stejnými závaznými pravidly pro práci.
Při každé smysluplné změně projektu aktualizuj **oba soubory**, aby se při přepínání mezi Codexem a Claude Code neztratil kontext.

Za smysluplnou změnu se bere hlavně: architektura, doménový model, modulový systém, API kontrakty, deploy postup, bezpečnostní pravidla, workflow vývoje, testovací brány a důležitá produktová rozhodnutí.

Veřejný ceník (`src/lib/pricing.ts` + `IndustryPackages.vue`) prodává jen ověřené workflow, která v aplikaci už běží, a označuje je štítkem „V aplikaci". Nezaměňuj připravený integrační cíl za ostrý provoz: klientská zóna umí faktury a přijetí nabídek, ne online platbu bez runtime provideru; cashflow připraví e-mailovou upomínku, neposílá automatickou sekvenci; BankID a platební provideri zůstávají „Připraveno k napojení", dokud nejsou aktivní adaptéry a credentials.

Předávací protokol zakázky je neměnný snapshot práce a materiálu. API detail vrací `handover.items` s `kind` `work`/`material`; `useJobs` musí detail defensivně normalizovat, aby neúplná starší odpověď nikdy neshodila stránku zakázky.

Nákupní objednávky V1 jsou přímo v existující stránce `/app/naskladneni`, nikoli v paralelním skladu: adresář dodavatelů → `Draft → Ordered → PartiallyReceived → Received`, s možností zrušení jen před prvním příjmem. UI `PurchaseOrdersPanel.vue` je mobilní, katalog načítá celý přes `loadAll()` a dílčí příjem drží stejný klientský UUID `idempotencyKey` při opakování po chybě. Backend atomicky znovupoužije existující `PurchaseReceipt` a append-only ledger. Používej jen stávající `inventory.read/manage`; role/oprávnění, nativní aplikace a AI/MCP v tomto toku neměň. Kontrakt: `docs/backend/purchase-orders-v1.md`.

Šarže a expirace INV-16 jsou mobilní webové rozšíření stejného skladu. Produkt má jednosměrné `lotTrackingEnabled`; aktivace v `SkladPage.vue` je celofiremní, přiřadí starou zásobu do systémové šarže a výslovně varuje, že sledovaný produkt už nesmí do záporného stavu ani přes POS/QR/výrobu. `NaskladneniPage.vue`, dílčí příjem objednávky a `ProductionBatchDialog.vue` posílají u sledovaných produktů číslo šarže a volitelnou expiraci. `ZasobyPage.vue` má tab `Šarže`: zůstatky/expirace, přesný odpis, ruční příjem a volitelný výběr šarže pro výdej/přesun; bez výběru používá backend FEFO. ## Co to je

Vystaveno.cz — fakturační + pokladní (POS) / gastro SaaS pro malé české firmy.
Tento repozitář = **frontend**: Vue 3 (`<script setup>` SFC) + Vite + TypeScript, package manager **npm**.
Migrace ze staré React appky (zazálohovaná ve větvi `legacy-react`, nevracet do working tree).

Produktová roadmapa modulárního Vystaveno + gastro priority je v `docs/product/modular-business-roadmap.md`. Při změně zásadního produktového směru ji aktualizuj společně s tímto kontextem.
Uživatelská nápověda: `docs/product/vystaveno-user-guide.md` je centrální krátký průvodce všemi obory; `docs/product/gastro-user-manual.md`, `sluzby-zakazky-manual.md` a `opakovane-faktury-manual.md` jsou detailní návody pro konkrétní workflow. Při každé uživatelsky viditelné změně aktualizuj příslušný návod. V aplikaci je `/app/pruvodce`: z `src/lib/help.ts` zobrazí podle zapnutých modulů a role jen relevantní krátké postupy; drž ji v souladu s navigací a těmito návody. `docs/product/interni-test-manual.md` je stručný průvodce pro interní test na VPS staging (gastro start, sklad/inventura, uzávěrka, integrace, podpisy) s tabulkou „co je ostré vs. jen připraveno k napojení" — drž ji pravdivou.
Claude Code continuation prompt pro samostatnou pipeline práci nad gastro roadmapou je v `docs/agent-workflow/claude-gastro-pipeline-prompt.md`; používej ho při předávání práce do Claude Code nebo po delší pauze.

Modulární runtime základ: backend `/me` vrací `modules` + `features`; frontend je drží v `src/stores/auth.ts` a app shell filtruje navigaci/routy přes `src/lib/modules.ts`. Backend ukládá tenant-selected moduly v `company_modules`; staré firmy bez řádků mají fallback na všechny default moduly. Onboarding vybírá první modulový balíček podle typu podnikání a ukazuje profilový checklist `setupSteps`; po uložení pošle firmu na první doporučený krok (Gastro začíná v `Pobočky`). `Nastavení firmy` umožňuje pozdější změnu modulů přes `/company/modules`. Modul `core` je povinný a normalizace ho vždy doplní. Stránky nesmí shodit celý view, když tenant nemá modul k části dat: `Přehled`/`DashboardPage.vue` volá fakturační část (metriky faktur, graf tržeb, poslední faktury/klienti + tlačítka `Nový klient`/`Nová faktura`) jen při `auth.hasModule('invoicing')` — bez modulu backend vrací 403 a dřív to shodilo celý Přehled jako „Server je momentálně nedostupný"; teď se fakturační část přeskočí a skryje, POS/gastro provoz zůstává. Gatuj přes `computed(() => auth.hasModule('X'))`. E2E `e2e/dashboard-modules.spec.ts`. Ze stejného důvodu je stránka i nav `Klienti` gate-ovaná modulem `invoicing` (ne `core`): klient je fakturační entita a backend `/clients` gatuje `Permissions.Invoices.Read → ProductModules.Invoicing`, takže bez modulu vracel 403 a `KlientiPage` ukazovala stejné „Server je momentálně nedostupný". Tenant bez invoicing `Klienti` v navigaci nevidí a `/app/klienti` route guard přesměruje na Přehled. E2E `e2e/klienti-modules.spec.ts`.

Fakturace — typy dokladu (Invoicing V2): modul `invoicing` umí běžnou fakturu, **zálohovou (proforma)** i **dobropis**. `DocumentType = 'invoice' | 'proforma' | 'credit_note'` + `Invoice.parentInvoiceId`. Editor vybírá Faktura/Zálohová (dobropis se v editoru netvoří); seznam má filtr/badge dle typu a akce „Vystavit dobropis" a „Převést na fakturu" (`useInvoices.creditNote`/`convertToInvoice`). **Dobropis dělá backend** (`POST /invoices/{id}/credit-note`, záporné částky/DPH server-side) — FE nefabrikuje záporné položky ani nepočítá DPH, `quantity` zůstává kladné (validator `> 0`), FE jen zobrazí zápornou odpověď. Dobropis vzniká rovnou vystavený (Issued) a needituje se (editor guard). Proforma (nedaňová) je vyloučená z DPH/pohledávek/obratu/dashboardu/účetnictví; daňový doklad vzniká konverzí (`POST /invoices/{id}/convert-to-invoice`). Dobropis je daňový doklad → v DPH a účetním exportu nettuje záporně; pohledávek ani obratu se netýká. Editor drží vztah `Label`/`id` pro odběratele a každé pole položky a ikonový přepínač náhledu má vždy `aria-label`; `e2e/a11y.spec.ts` tak hlídá serious/critical WCAG regresi celého editoru. Backend kontrakt: `docs/backend/invoicing-v2.md`. E2E `e2e/faktura-doklady.spec.ts`. **FE↔BE mapovací adapter (jen API režim):** invoicing FE byl původně psaný proti mock tvaru a nebyl namapován na backend; `src/lib/invoice-api.ts` překlápí backend DTO ↔ FE `Invoice` (`invoiceFromApi`/`invoiceToCreateRequest`/`invoiceToUpdateRequest`): enum casing (`Draft|Issued|Paid|Cancelled|Sent|Overdue|Archived`→lowercase, `Invoice|Proforma|CreditNote`→`invoice|proforma|credit_note`, idempotentně), `postalCode`↔`zip`, `bic`↔`swift`, `supplierName`→`companyName`, `isVatPayer`→`vatMode`, `lineBase`→`lineSubtotal`; FE peníze/DPH nepočítá, jen přemapuje serverová čísla (znaménka dobropisu zachová). Aplikuje se VÝHRADNĚ v API režimu ve `useInvoices` (mock beze změny). `useInvoices.pay` → `POST /invoices/{id}/mark-paid` s `{ idempotencyKey }` (ne neexistující `/pay`). `GET /invoices` je summary bez řádků → detail/editace přes `useInvoices.get(id)` = `GET /invoices/{id}`. **Fakturace dokončená v API režimu (GAP 1+2):** editace řádků konceptu se po PUT hlavičky synchronizuje přes `/items` — čistý diff `diffInvoiceLines(serverIds, editorItems)` (v `invoice-api.ts`, unit-testovaný), pak bezpečné pořadí `PUT /items/{id}` úpravy → `POST /items` nové **PŘED** mazáním (draft nespadne na 0 řádků) → `DELETE /items/{id}` → `PUT /items/reorder` finální množinou server-id v pořadí editoru (klientské uuid → server id z POST odpovědi). **DphPage** bere v API režimu DPH z `GET /invoices/vat-summary?from=&to=` (`useInvoices.vatSummary`, tvar `VatSummary`; server nettuje dobropis a vylučuje proformu; 'YYYY-MM'→měsíční rozsah, 'all'→bez); mock režim dál `dph.ts` client-side. **Účtárna** CSV vystačí s hlavičkovými součty (list-summary nese `subtotal`/`vatTotal`), ISDOC v API režimu dotáhne detail `get(id)` (kvůli řádkům) a pak `downloadIsdoc`. ## Dva repozitáře, jeden produkt

- **`vystavenocz`** (tento) — frontend (Vue).
- **`vystaveno-api`** (samostatný repozitář) — backend: .NET / ASP.NET Core + PostgreSQL + EF Core. Při vývoji leží vedle jako `../vystaveno-api`.

Frontend volá backend přes **same-origin `/api/v1`** (žádné CORS).
Datový kontrakt: `src/lib/types.ts` (FE) musí sedět s DTO na backendu; sdílené kontrakty jsou popsané v `docs/backend/*.md`.
**Když měníš tvar přenášených dat, uprav OBĚ strany** (FE typy + BE DTO) a případně kontrakt v `docs/backend/`.

Skladová karta INV-16: `Zásoby` → `Pohyby` používá serverově stránkované `useInventory.movementPage()`, úplný export `movements()` a filtrační katalog `/inventory/movement-filters`; kontrakt je v `docs/backend/inventory-ledger-export.md`. Klient musí pro přesný CSV export načíst všechny stránky, zachovat `related*Id` a při změně `total` během načítání skončit bezpečně bez souboru. `productName`, `productSku` a `locationName` jsou aktuální popisky; autoritativní jsou stabilní ID. Nativní mobilní aplikace a oprávnění nejsou součástí tohoto řezu.

## Deploy (produkce = VPS only)

Produkce běží na vlastním VPS (OVH), Docker stack: Caddy (TLS) → nginx (SPA statika + reverse-proxy `/api`) → .NET API → PostgreSQL.
Deploy soubory jsou v TOMTO repu (`Dockerfile`, `docker-compose*.yml`, `nginx.conf`, `Caddyfile`); API se builduje ze sousedního `../vystaveno-api` (musí ležet vedle).

Deploy je **pull-based** — po merge do `main` se samo NIC nenasadí. Uživatel to pouští ručně na VPS:

```bash
cd ~/vystavenocz && git pull
cd ~/vystaveno-api && git pull
sudo sysctl -w net.ipv6.conf.all.disable_ipv6=1   # VPS nemá IPv6, jinak build API padá na stahování
cd ~/vystavenocz && docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
sudo sysctl -w net.ipv6.conf.all.disable_ipv6=0
```

`Dockerfile` zapéká `ARG VITE_API_URL=/api/v1` do buildu → přebíjí `.env.production` (ten je pro VPS irelevantní, needituj ho kvůli produkci).
VPS `.env` musí obsahovat i `PAYMENTS_PORTAL_BASE_URL` (veřejná URL aplikace, typicky `https://$DOMAIN`), jinak online checkout faktury nemá kam vrátit zákazníka.
Migrace databáze naskočí samy při startu API. Ověření živého stavu: `curl https://$DOMAIN/health/ready` → 200 „Healthy".
Po deployi projdi `docs/deploy-smoke-checklist.md` (health/live, health/ready, `/api/v1/ping`, přihlášení, moduly, POS prodej, restaurace/stůl, kuchyně, uzávěrka, sklad/inventura/zrcadlo, Pohoda XML export, tiskový agent, platební provider settings + vault, podpisy provider settings + vault + odeslání, veřejné/QR menu a audit). Před deployem ověř zálohu DB; migrace jsou forward-only.
Veřejná landing page na `/` musí při VPS deployi zůstat zachovaná; aplikace běží pod `/app` a API pod `/api`. Deploy smoke kontroluje `/` zvlášť, aby se homepage omylem nepřepsala.
Produkční spolehlivost V1: deploy spouštěj přes `ops/vps-deploy.sh`, který sdílí `flock` s `ops/vps-backup.sh`, před změnou vytvoří konzistentní PostgreSQL + `api_files` balík a čeká na health. `ops/vps-verify-backup.sh` obnovuje dump jen v izolovaném jednorázovém PostgreSQL; `ops/vps-health-check.sh` hlídá endpointy, Compose služby, stáří zálohy a disk. Instalace, retence, off-site hranice a recovery jsou v `docs/vps-reliability.md`. `.ops.env` je server-only (mode 600, gitignored). Neměň `INTEGRATIONS_SECRET_ENCRYPTION_KEY` jako rychlou opravu.
Staging demo data: backend `vystaveno-api` má explicitní CLI helper `dotnet Vystaveno.Api.dll seed-demo`, který se spouští jednorázově přes Docker exec s `Seed__AllowDemo=true` (viz `docs/deploy-smoke-checklist.md`). Naplní demo firmu `Vystaveno Demo Gastro` pro interní smoke test; login je `demo@vystaveno.cz` / `DemoGastro.2026`. Demo **simuluje cca rok provozu napříč všemi moduly** (roční historie prodejů přes 3 pobočky, uzavřené dny/Z-reporty za rok + měsíční exporty, fakturace za rok se stavy zaplaceno/po splatnosti/vystaveno, publikované směny + odpracovaná docházka, rezervace ve víc stavech, šablona opakovaných faktur s historií běhů), aby reporty/přehledy/uzávěrky nebyly prázdné. Backend seeder žije v `DemoDataSeeder*.cs`; roční historie prodejů se backdatuje jedním hromadným UPDATE (`unnest`), historické Z-reporty se dopočítají z denní agregace prodejů. **Roční data vzniknou jen při PRVNÍM seedu firmy** — seeder je idempotentní přeskočením, takže na už existující demo firmě opakovaný běh jen dorovná chybějící moduly; pro bohatší demo se stávající demo firma musí nejdřív smazat a naseedovat znovu (izolovaná ne-produkční data). Sklad zůstává na snapshotu (roční skladové pohyby vědomě mimo scope). Není to běžná produkční funkce, nespouští se při startu aplikace a nespouštěj ho na reálné produkční databázi.

- Koncové rozhraní používá běžnou češtinu. Interní pojmy jako backend, runtime, adapter, credential, scope, mock nebo název proměnné prostředí do UI nepatří.
- Preferované názvy v navigaci jsou `Stoly a objednávky`, `Kuchyňské objednávky`, `Produkty a menu`, `Stav skladu`, `Příjem zboží`, `Volby k produktům`, `Denní uzávěrka` a `Propojení pro vývojáře`.
- Technické termíny ponechávej jen v pokročilé části pro vývojáře a hned je vysvětli běžnými slovy.
- Falešné úspěchy jsou zakázané: ukázková akce nesmí tvrdit, že odeslala e-mail, strhla platbu nebo provedla právní podpis.
- Zdroj kompletní příručky je `docs/product/kompletni-uzivatelsky-manual.md`. PDF generuje `scripts/generate-user-manual-pdf.py`; po změně zásadního workflow aktualizuj zdroj i PDF.
